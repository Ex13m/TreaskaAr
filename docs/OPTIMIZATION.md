# Оптимизация: методология и бюджеты

Цель из ТЗ: **60 FPS на среднем телефоне** при 14 рыбах + 20 снарядах +
3 взрывах. Подход — «оптимизация без потери функционала»: сначала меряем,
потом убираем стоимость, не трогая то, что видит и чувствует игрок.

## Бюджет кадра (16.6 мс)

| Статья | Цель | Чем обеспечено |
|---|---|---|
| Дроуколлы | < 40 | инстансинг рыб (5), пулы эффектов, один PointLight |
| JS-логика | < 3 мс | плоские данные, ноль аллокаций в кадре |
| Растеризация | плавающая | динамическое разрешение ×1…×0.6 |
| GC-паузы | 0 | пулы всего, переиспользуемые `_v1.._v3`, `_dummy` |

Ориентир по дроуколлам — индустриальный: до ~100 на кадр для мобильного
WebGL; инстансинг сокращает их на 90%+ в сценах с массой однотипных
сущностей. Проверяется через `?debug` (см. ниже).

## Применённые техники (и где они в коде)

1. **GPU-инстансинг + запекание** (`bakeParts`, `initFishSystem`).
   Вид рыбы = одна геометрия с вертексными цветами = один `InstancedMesh`.
   Было: ~10 мешей × 14 рыб ≈ 140 дроуколлов. Стало: **5**.
   Слоты переиспользуются через free-list, смерть = нулевая матрица.

2. **Анимация в вершинном шейдере** (`fishMaterial`).
   Виляние хвостом — `sin(aPhase + z·k)` с маской по длине тела.
   CPU пишет один float на рыбу за кадр вместо вращения иерархии мешей.
   Правило: всё, что можно посчитать на вершинах, не считаем на CPU и
   не считаем во фрагментах.

3. **Один скомпилированный шейдер на все виды.**
   GLSL-инжект одинаков, различия видов — юниформы `uAmp`/`uLen`.
   Нет лишних компиляций и переключений программ.

4. **Пулы объектов** — снаряды (32), вспышки частиц (10×36), кольца (6),
   воксели (48), всплывающие цифры (14 DOM-узлов), стрелки радара (3),
   ореолы золотых (4). Аллокаций в горячем пути нет; временные векторы —
   модульные синглтоны `_v1.._v3`.

5. **Динамическое разрешение** (`qualityTick`).
   Окно 2 с: FPS < 50 → масштаб −0.15 (минимум 0.6); FPS > 57 → +0.1.
   Стартовый кап pixelRatio = 2 (по ТЗ), рабочий диапазон на телефонах
   фактически 1.2–2 — в духе рекомендации держать 1–1.5 на слабых GPU.
   Деградирует только разрешение канваса: геймплей, эффекты, UI — нетронуты.

6. **Дешёвый «блум» вместо постпроцессинга.**
   Аддитивные спрайты-глоу + PointLight на 2 кадра дают неон без
   полноэкранных проходов, которые мобильный GPU не прощает.

7. **Свет и материалы.** Lambert вместо PBR, ноль теней, два источника
   света. Эмиссив-пульс золотых/мигание фугу — на общем материале вида
   (одна запись юниформа, не перебор объектов).

8. **DOM-гигиена.** HUD — DOM (дёшево и чётко), но: ссылки на элементы
   кэшируются (`chipEls`), всплывающие числа анимирует CSS-транзишн
   (компоузер), а не JS.

## Как профилировать

- **`?debug`** в URL — оверлей: fps, calls, tris, масштаб рендера, рыбы.
  Смоук-тест: в разгар «золотой лихорадки» calls < 40, fps ≥ 55.
- Chrome DevTools → Performance с CPU throttling ×4 + про эмуляцию тача.
- На устройстве: `chrome://inspect` (Android) / Web Inspector (iOS).
- Память: Performance Monitor → JS heap не должен «пилить» (пила = аллокации
  в кадре = будущие GC-фризы).

## Чему говорили «нет» (и почему)

- **UnrealBloomPass/постпроцессинг** — 2–3 полноэкранных прохода съедают
  весь мобильный бюджет; эффект достигнут аддитивными спрайтами.
- **Тени** — запрещены бюджетом и не читаются на фоне видеокамеры.
- **SkinnedMesh для рыб** — кости избыточны для синусоиды хвоста.
- **Физический движок** — сферические коллизии и steering покрывают всё.
- **WebGPU сейчас** — выигрыш есть, но парк iOS-устройств ещё неровный;
  отмечено в роадмапе README как следующий шаг с фолбэком на WebGL.
